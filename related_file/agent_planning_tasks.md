# LangChain4j 规划智能体实施任务清单

## 项目目标

让系统能够识别复杂问题，生成任务计划，分步调用工具执行，并最终汇总出清晰答案。

当前定位：

- 保留现有普通 RAG 问答链路
- 使用 LangChain4j 构建复杂问题的 Planning Agent
- 使用本地 Ollama 作为最终答案生成模型
- 先做受控智能体，不做自由自治 Agent

---

## 总体实施原则

1. 普通 RAG 问答先不动
2. LangChain4j 只先接管复杂问题链路
3. 先做受控 Agent，不做自由自治 Agent
4. 先跑通一个复杂场景，再扩展更多场景

---

## 任务 0：冻结当前可用版本

### 目标
先把当前“普通 RAG + Ollama 最终答案生成”版本固定住，避免后续引入 LangChain4j 时影响现有链路。

### 要做的事
- 新建独立开发分支
- 确认当前普通问答链路可用
- 记录当前接口、主要服务类、调用链
- 准备一组固定的回归测试问题

### 完成标准
- 普通问答链路可稳定回答
- 有一组回归测试问题可重复执行
- 可以随时回退到当前版本

## 任务 1：引入 LangChain4j 基础依赖

### 目标
将 LangChain4j 接入现有 Spring Boot 项目，但不替换现有 RAG 主链路。

### 要做的事
- 增加 LangChain4j 相关依赖
- 配置 Ollama 对接 LangChain4j
- 新建 agent 相关基础包结构
- 验证项目可正常启动
- 验证 LangChain4j 能调通本地 Ollama

### 你要产出的东西
- agent 模块基础目录
- LangChain4j 基础配置类
- 一个最小连通性验证

### 完成标准
- 项目能正常启动
- LangChain4j 能调用本地 Ollama
- 不影响现有普通问答接口

---

## 任务 2：定义 Agent 的领域模型

### 目标
先把智能体的基础对象设计好，作为后续规划、执行、记录的统一数据结构。

### 要做的事
定义下面这些核心对象：

- `AgentTaskType`
- `AgentComplexity`
- `AgentPlan`
- `AgentStep`
- `AgentExecutionContext`
- `AgentStepResult`
- `AgentFinalResponse`

### 重点
这些对象要能完整表达：
- 原始问题
- 复杂度判断结果
- 任务计划
- 每一步执行结果
- 最终答案
- 执行摘要

### 完成标准
可以用这些对象完整描述一条复杂任务的处理过程。

---

## 任务 3：实现问题复杂度判断

### 目标
先把问题区分成两类：

- `SIMPLE`
- `COMPLEX`

### 要做的事
实现 `ComplexityJudgeService`，采用“**规则优先 + 预留 LLM 扩展**”的方式。

### 第一版规则建议
满足任一条件就倾向 `COMPLEX`：
- 出现“对比 / 分析 / 总结 / 归纳 / 区别”  **等多内容**的词
- 出现**多个对象**，如“三篇论文 / 多篇文章 / 多个文档”   
- 同时要求**多个维度**，如“方法 + 实验结果”
- 明显不是一次检索就能回答

### 完成标准
至少能正确区分：
- “这篇论文的方法是什么” → SIMPLE
- “帮我对比三篇论文的方法和实验结果” → COMPLEX

---

## 任务 4：定义 Agent 可用工具集合

### 目标
先确定智能体可调用的工具，避免后续执行链失控。

### 第一版只做这 5 个工具
- `retrieveChunks`
- `extractMethod`
- `extractExperimentResults`
- `compareFindings`
- `summarizeFinalAnswer`

### 要做的事
- 定义统一的工具接口
- 定义工具注册表
- 将现有 `RetrievalService`、`LlmService` 等能力包装为工具

### 完成标准
Agent 可以通过统一入口调用工具，而不是在业务代码中到处写 if-else。

---

## 任务 5：实现 Planner

### 目标
让复杂问题先生成任务计划，而不是直接回答。

### 要做的事
实现 `PlannerService`。

第一版不要自由规划，先做受控规划：
- 只支持“多论文对比”这一类任务
- 输出固定结构的 plan

### 针对示例问题的固定拆解
问题示例：
> 帮我对比三篇论文的方法和实验结果

固定拆分为：
1. 检索目标论文相关内容
2. 提取每篇论文的方法
3. 提取每篇论文的实验结果
4. 比较差异
5. 生成总结

### 完成标准
输入复杂问题时，系统能输出结构化计划对象，而不是随意文本。

---

## 任务 6：实现 StepExecutor

### 目标
把 plan 真正执行起来。

### 要做的事
实现 `StepExecutorService`，负责：
- 按顺序执行步骤
- 根据 `step.action` 查找对应工具
- 将每一步结果写入执行上下文
- 失败兜底
- 限制最大步数与最大补充检索次数

### 第一版限制
- 最多 5 步
- 最多 2 次补充检索
- 某一步失败最多重试 1 次
- 不允许无限循环

### 完成标准
可以完整看到每一步：
- 执行了什么
- 输入是什么
- 输出是什么
- 最终如何形成答案

---

## 任务 7：把 Agent 和现有普通问答链路接起来

### 目标
形成统一入口，但内部自动分流。

### 要做的事
实现 `AgentService` 或统一编排服务：

#### 流程
- 先判断复杂度
- SIMPLE → 走现有普通 RAG
- COMPLEX → 走 Planner + Executor

### 重点
Agent 是新增一层，不是推翻现有问答主链路。

### 完成标准
同一个系统入口下：
- 简单问题仍然正常直答
- 复杂问题开始规划执行

---

## 任务 8：加执行链记录

### 目标
让智能体过程可观察、可解释、可调试。

### 要做的事
记录以下内容：
- 用户原始问题
- 复杂度判断结果
- 生成的计划
- 每一步调用了什么工具
- 每一步输入输出摘要
- 最终答案
- 总耗时
- 失败原因

### 第一版实现方式
先不用落数据库，先用：
- 日志
- 内存 trace 对象
- 接口返回 `traceSummary`

### 完成标准
可以在调试时明确知道：
- 是规划错了
- 还是工具执行错了
- 还是总结出了问题

---

## 任务 9：对外暴露 Agent 接口

### 目标
给前端或 Postman 提供明确的 Agent 调用入口。

### 要做的事
新增最小接口，例如：
- `POST /api/agent/ask`

### 请求体至少包含
- `question`
- `knowledgeBaseIds`（可选）

### 返回至少包含
- `complexity`
- `finalAnswer`
- `plan`
- `traceSummary`

### 完成标准
可以通过接口直接测试复杂问题链路。

---

## 任务 10：先做一个唯一复杂场景的验收

### 目标
先只打磨一个复杂场景，不同时做多种复杂任务。

### 第一版指定场景
**多论文对比**

### 典型问题示例
- 帮我对比三篇论文的方法和实验结果
- 总结这几篇论文的共同点和差异
- 比较这三篇文章在实验设计上的不同

### 要做的事
准备 5 到 10 个测试问题，验证：
- 是否能正确判断为复杂问题
- 是否能生成合理计划
- 是否能按步骤执行
- 是否能输出结构化答案

### 完成标准
这个场景可以稳定跑通。

---

## 任务 11：做失败兜底和边界控制

### 目标
防止智能体跑飞，保证可控性。

### 要做的事
增加以下保护：
- 计划为空时降级为普通 RAG
- 某一步连续失败时停止执行
- 检索结果不足时给出信息不足提示
- 最终总结前检查中间结果是否为空
- 超过最大步数直接终止

### 完成标准
系统不会因为某一步异常而无限重试或直接崩掉。

---

## 任务 12：最后再考虑前端展示

### 目标
让用户能明显感知系统是在“先规划再执行”。

### 第一版前端建议展示 4 块
- 问题复杂度
- 执行计划
- 执行步骤
- 最终答案

### 注意
先不要追求花哨页面，重点保证：
- plan 看得见
- step trace 看得见
- final answer 看得见

### 完成标准
用户可以直观看到智能体执行轨迹。

## 第一阶段最终里程碑

完成后，你应该得到一个真正有价值的 MVP：

**系统能够识别复杂知识任务，并基于计划进行多步检索与执行，再汇总出最终答案。**

这时你的系统已经不是普通 RAG，而是具备初步规划能力的知识库智能体。
